# 开发者站 UI / 品牌落地规范

> 用途：确保 `autional-dev-site` 后续新增页面时，继续沿用已落地的品牌体系，而不是退回零散样式拼接。

---

## 1. 品牌目标

开发者站的视觉目标应保持：

- 专业
- 克制
- 企业级
- 兼顾品牌感与技术密度

开发者站不是营销页，但也不能只是裸露的技术文档列表。

---

## 2. 基础视觉

应优先复用：

- `primary`
- `sky`
- `amber`
- `shadow-soft`
- `shadow-card`
- `shadow-brand`

避免：

- 零散硬编码颜色
- 随意增加新的蓝色系
- 与主官网不一致的阴影和圆角

---

## 3. 全局品牌类

优先复用以下类：

- `brand-shell`
- `brand-card`
- `brand-button-primary`
- `brand-button-secondary`
- `brand-kicker`
- `brand-grid`
- `developer-panel`
- `developer-code`
- `developer-chip`
- `developer-nav-link`

若新增开发者站专用视觉组件，优先补充到：

- `src/styles/global.css`

---

## 4. 页面结构规范

开发者站关键页面建议遵循：

- 顶部有明确标题与用途说明
- 页面中有清晰的信息分组
- 有足够留白和卡片层次
- 结尾有收束区或下一步入口

不要让页面退回成：

- 大段文本直接堆叠
- 只有边框盒子，没有信息层级
- 只有列表，没有导航和引导

---

## 5. 代码块规范

代码块应：

- 优先使用深色面板
- 保证足够对比度
- 允许横向滚动
- 对长命令和示例代码保持可读性

如果代码内容包含 JSX / HTML 风格标签：

- 优先使用普通文本渲染
- 不要使用会把内容当作真实 HTML 的渲染方式

---

## 6. Header / Footer 规范

Header：

- 承担主要品牌识别
- 使用图形标 + 品牌文字
- 导航应清晰可见

Footer：

- 只做信息收束与次级导航
- 不需要过度强化图形 logo
- 保持克制，不抢主内容

---

## 7. 内容型页面规范

Quickstart、Migration、API 这类页面应：

- 优先采用卡片或步骤结构
- 明确“先做什么，再做什么”
- 让用户能快速抓住行动路径

避免把文稿原样塞进页面而不做结构化处理。

---

## 8. 技术规范

- 品牌令牌统一在 `tailwind.config.cjs`
- 共享视觉类统一在 `src/styles/global.css`
- 共享壳层统一在 `src/layouts/DevLayout.astro`
- 新页面优先复用已有类，而不是重复拼 Tailwind 原子类

---

## 9. 后续维护建议

新增或改动页面前，先检查：

1. 是否复用了已有品牌类？
2. 是否与首页和共享布局风格一致？
3. 是否保证代码块可读？
4. 是否给开发者提供了明确下一步动作？

如果以上问题大多回答为“否”，建议先回到本规范重新设计。
