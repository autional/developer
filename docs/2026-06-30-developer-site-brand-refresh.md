# 开发者站品牌化改造详细变更说明

> 记录日期：2026-06-30  
> 项目：`autional-dev-site`  
> 变更类型：品牌视觉升级、共享布局重构、核心页面重做、品牌资源接入  
> 适用范围：开发者站首页、Quickstart、Migration、API 页面、共享布局、全局样式与品牌资源

---

## 1. 目标是什么

本次变更的目标，是让 `autional-dev-site` 在视觉上与主官网 `autional-web` 保持一致的品牌语言，同时保留“开发者站”所需要的技术表达密度。

具体目标包括：

1. 让开发者站不再像默认文档模板或简陋的产品子站。
2. 让 Header、Footer、Logo、配色、按钮、卡片与主官网保持同一品牌体系。
3. 让首页更像成熟 SaaS 的开发者门户，而不是简单的包列表。
4. 把 `quickstart` 从近似 Markdown 草稿页升级成可浏览、可引导的正式页面。
5. 为后续新增开发者页面沉淀可复用的样式基础与规范文档。

---

## 2. 变更前存在什么问题

### 2.1 整体视觉偏模板化

改造前的开发者站具备基础可用性，但视觉表达存在明显短板：

- 共享布局非常基础，只有一条简易 Header。
- 没有统一的品牌外壳，站点气质和主官网断层。
- 页面结构更多是“信息堆叠”，缺少开发者品牌门户感。

### 2.2 样式缺少统一基础设施

项目中原本没有全局品牌样式层：

- 没有 `global.css`
- 没有统一按钮、卡片、容器类
- 没有品牌级配色、字体和阴影令牌

结果是页面样式只能依赖零散的 Tailwind 原子类直接拼接，后续维护成本高，视觉也容易漂移。

### 2.3 首页信息表达过于朴素

首页虽然列出了 SDK 包和 API 客户端，但问题是：

- 没有明确的 Hero 区域
- 没有主次 CTA
- 没有品牌化代码面板
- 缺少“这是专业开发者平台”的第一眼识别

### 2.4 Quickstart 页面不是真正的站点页面

`quickstart.astro` 原始内容实际上更像直接粘进去的 Markdown 文稿：

- 没有共享站点风格
- 没有卡片式步骤结构
- 没有视觉引导
- 与其他页面完成度严重不一致

### 2.5 品牌资源未统一

改造前该站点没有接入和主官网一致的品牌 `logo-mark.svg` 与 `favicon.svg`，导致品牌识别不连续。

## 3. 实际做了什么

### 3.1 建立品牌样式基础设施

新增文件：

- `src/styles/global.css`

在其中沉淀了开发者站可复用的品牌类：

- `brand-shell`
- `brand-card`
- `brand-button-primary`
- `brand-button-secondary`
- `brand-kicker`
- `brand-grid`
- `developer-panel`
- `developer-code`
- `developer-chip`
- `developer-nav-link`

这样做的原因是：

- 避免每个页面重复拼接 Tailwind 类
- 让站点在未来继续扩展时能够沿用同一视觉基础
- 把“这次美化”变成“长期可复用的样式资产”

### 3.2 扩展 Tailwind 品牌令牌

更新了：

- `tailwind.config.cjs`

补齐了与主官网一致的品牌主题：

- `primary`
- `sky`
- `amber`
- 字体栈
- 阴影层级

采用这种方式的原因：

- 保证品牌色的统一来源
- 减少未来页面出现硬编码颜色值的概率
- 使后续新增页面更容易复用样式

### 3.3 重构共享布局

更新了：

- `src/layouts/DevLayout.astro`

具体改动包括：

- 引入全局样式
- 接入品牌字体
- 重做 Header
- 新增 Footer
- 在 Header 中接入品牌 Logo 与更完整导航
- 在 Footer 中使用更克制的文字品牌收口

这样做的目的，是让每个页面无需单独处理品牌壳层，统一由布局文件承接。

### 3.4 重构首页

更新了：

- `src/pages/index.astro`

具体变化包括：

- 新增 Hero 区
- 新增主次 CTA
- 新增统计卡片
- 新增右侧深色代码面板
- 重新组织 SDK 包展示区
- 增加“Generated Clients” 区块
- 增加底部 CTA 收束

采用这种结构的原因：

- 开发者站首页不能只是“包列表”，需要先建立平台气质
- Hero + 指标卡 + 代码面板，比纯列表更能传达产品成熟度
- 这样的信息架构也更接近主官网的品牌表达节奏

### 3.5 重做 Quickstart 页面

更新了：

- `src/pages/quickstart.astro`

这部分改造是本次比较重要的内容，因为原始页面完成度明显不足。

重做后：

- 将内容转为正式 Astro 页面
- 增加三步卡片式引导
- 补足 prerequisites 区域
- 增加 “What You Get” 与 “Common Issues” 区域
- 保留原有核心信息，但重新组织了视觉层次

这样做的原因：

- Quickstart 是开发者站的重要入口
- 如果它看起来像草稿，会直接降低用户对平台可靠性的感知

### 3.6 优化 API 与 Migration 页面

更新了：

- `src/pages/api.astro`
- `src/pages/migration.astro`

具体做法：

- 使用统一品牌卡片样式
- 加入更加明确的标题层次
- 让信息块从“普通段落+框”升级为“更有结构的技术展示卡片”

### 3.7 接入品牌资源

新增资源：

- `public/logo-mark.svg`
- `public/favicon.svg`

资源来源与主官网一致，使开发者站与主站的品牌资产保持同步。

## 4. 为什么采用这些方式

### 4.1 为什么优先做布局和样式层

如果直接只改首页，其他页面仍然会显得旧、散、弱，整体完成度不高。

因此本次优先级是：

1. 样式令牌与全局组件类
2. 共享布局
3. 首页与核心页面

这样改完后，不是单个页面变漂亮，而是整站基础完成度一起提升。

### 4.2 为什么 Quickstart 要重做而不是只加类名

原页面不是“样式不够好”，而是信息组织方式本身就像文稿。

仅补几个类名无法解决：

- 节奏感不足
- 引导性不足
- 页面完成度不够

所以采用“保留内容、重构结构”的方式更合理。

### 4.3 为什么 Footer 采用克制方案

延续主官网经验，Footer 不再追求强品牌符号展示，而是：

- 承接信息
- 做页面收口
- 不和 Header 抢品牌识别

这种方案更适合长期维护，也更稳妥。

## 5. 实施过程中遇到的困难

### 5.1 项目原本没有全局样式层

这意味着品牌化不是“改现有体系”，而是需要先从零补基础设施。

解决方式：

- 新建 `src/styles/global.css`
- 在布局中统一接入
- 将品牌容器、按钮、卡片、代码块做成复用类

### 5.2 需要验证但不希望把验证副产物带入提交

为了验证本次改动，项目安装了依赖并执行了 `npx astro check`。

过程中会产生：

- `package-lock.json`

但该仓库原本使用的是 `pnpm-lock.yaml`，因此这些临时副产物不应进入提交。

解决方式：

- 验证完成后删除无关的 `package-lock.json`

### 5.3 本地预览受沙箱限制

尝试启动 dev server 时，当前运行环境对 `.astro/types.d.ts` 的写入有限制，导致无法在此环境中直接打开实时预览。

解决方式：

- 转为使用 `npx astro check` 做静态正确性验证
- 将该限制记录为环境问题，而非项目代码问题

## 8. 新发现的技术债

### TD-1：移动端导航仍是简化状态

当前 Header 在小屏下没有独立的抽屉导航或移动端菜单按钮。

影响：

- 页面在移动端仍然可看，但导航完成度不够高

建议：

- 后续单独补一版移动端开发者站导航方案

### TD-2：代码块暂无更细粒度语法高亮

当前开发者站代码块已具备品牌化深色面板和可读性，但仍未接入正式语法高亮系统。

影响：

- 可读性已达标，但精致度仍有提升空间

建议：

- 后续可引入更细的 token 级高亮方案

## 9. 经验教训

### Lesson-1：开发者站也必须有品牌首页

开发者站不应该只是功能列表或 SDK 列表。  
它同样承担：

- 品牌表达
- 产品可信度建立
- 路径引导

### Lesson-2：文稿型页面要从结构上重做

当页面像草稿时，只加样式不会真正解决问题。  
应该从信息架构上调整，再配合视觉样式升级。

### Lesson-3：验证副产物必须及时清理

为了通过检查安装依赖是合理的，但不应该把与项目既有包管理策略不一致的锁文件混入提交。
